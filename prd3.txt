PRD completo considerando os 4 MVPs como escopo do sistema completo**. A lógica é: **construir o Aquimaq CRM como sistema completo de gestão operacional, gerencial e decisória**, mantendo o ERP legado como fonte oficial transacional/fiscal/financeira enquanto for necessário.

Base conceitual: organizar a empresa pelo padrão novo, migrando por ondas, com centros de resultado, cadastro mestre, estoque endereçado, compras por regra, oficina formalizada, DRE gerencial, painel semanal e indicador de `% da operação no padrão novo`. 

# PRD Completo — Aquimaq CRM

## Controle, Rotina, Resultado e Melhoria

Versão: 1.0
Produto: Aquimaq CRM
Stack recomendada: React + Vite + TypeScript + Supabase Cloud (n8n em VPS Hostinger — **fase posterior**, após o núcleo web estabilizado)
ERP legado: permanece como sistema oficial enquanto necessário
Escopo: sistema completo com os 4 MVPs incluídos desde o planejamento inicial

---

# 1. Resumo executivo

O **Aquimaq CRM** será o sistema interno completo para reorganizar a operação da Aquimaq, controlar processos, auditar cadastros, estruturar estoque, compras, recebimento, oficina, tarefas, indicadores e painel gerencial econômico, financeiro e contábil.

Apesar do nome CRM, nesta etapa o significado prático é:

```txt
C = Controle
R = Rotina
M = Melhoria
```

O sistema não nasce como CRM comercial de cliente. Ele nasce como uma plataforma para **organizar a casa, medir resultado e orientar decisão**.

A tese central:

```txt
ERP legado registra a operação oficial.
Aquimaq CRM organiza, audita, mede, cobra e melhora a operação.
```

O sistema completo deve responder diariamente:

```txt
o que está fora do padrão?
quem precisa corrigir?
qual é a prioridade?
onde estamos perdendo margem?
onde falta estoque?
qual compra precisa ser feita?
qual OS está parada?
qual recebimento deu problema?
qual centro de resultado está performando?
tem caixa?
o que está oficialmente registrado?
qual ação precisa acontecer esta semana?
```

# 2. Problema

A empresa está operando com acúmulo de vícios operacionais:

```txt
cadastros ruins
categorias confusas
estoque incerto
produtos sem localização
baixa rastreabilidade
compras reativas
margem pouco confiável
oficina informal
recebimento sem padrão forte
decisões sem indicadores
responsabilidades difusas
rotina de gestão fraca
dependência de conhecimento informal
```

Trocar de ERP ou manter o ERP legado não resolve sozinho esse problema. O problema principal não é apenas sistema. É **processo, dado, governança, rotina e decisão**.

O Aquimaq CRM será criado para atacar esse núcleo.

# 7. Princípios do sistema

## 7.1 Tudo deve virar ação

```txt
Toda informação relevante deve poder gerar tarefa, alerta ou decisão.
```

## 7.2 Operação primeiro, cliente depois

Nesta etapa, o foco não é CRM comercial. O foco é organizar a operação interna.

## 7.3 Painel não é enfeite

O painel deve mostrar problemas acionáveis.

```txt
Não basta mostrar número.
Precisa mostrar risco, responsável e próxima ação.
```

## 7.4 Dados oficiais continuam no ERP

O Aquimaq CRM pode importar, enriquecer e analisar dados, mas não deve conflitar com o ERP.

## 7.5 Separar econômico, financeiro e contábil

```txt
Econômico = lucro e margem.
Financeiro = dinheiro e prazo.
Contábil = registro oficial.
```

# 24. Modelagem de banco

## Abordagem — migrações e convenções físicas

### Migrações (`supabase/migrations/`)

- Ficheiros versionados: prefixo `YYYYMMDDHHMMSS` + sufixo descritivo em `snake_case`.
- **Uma fase do roadmap não implica um único ficheiro:** podem existir várias migrações por fase (schema, RLS, seeds, triggers) para rastreio fino e alterações mais seguras.
- **Correcções e alinhamentos** podem ter migração própria sem número de fase no nome (`hotfix_*`, `align_*`, etc.).
- O histórico aplicado no Supabase remoto pode divergir dos ficheiros locais após `migration repair` ou aplicação via MCP — alinhar com `supabase db pull` / comparar `supabase_migrations` quando o ambiente não bater com o Git.

### Nomenclatura

- **Tabelas:** `snake_case`; preferência por **plural** (`products`, `purchase_orders`). Exceções aceites: `profiles` (singular histórico), tabelas com prefixo de domínio (`erp_products`, `management_*`).
- **FKs:** padrão `<referência>_id` (`brand_id`, `result_center_id`, `work_order_id`). Nomes compostos mantêm legibilidade (`erp_product_id`, `responsible_user_id`).

### Soft delete e estado

- **Padrão actual no CRM:** inactivação com **`is_active`** (boolean) ou estados equivalentes; não se exige `deleted_at` em todas as entidades legadas.
- **`deleted_at`:** usar em novas mestras ou refactors quando o requisito for apagamento lógico auditável por data; não é requisito retroactivo universal.

### Campos padrão (por tipo de tabela)

- **Base mínima:** `id uuid` PK, `created_at timestamptz`, `updated_at timestamptz` (onde fizer sentido).
- **Mestras / operações sensíveis (novas ou em refactor):** acrescentar `created_by uuid`, `updated_by uuid` (referência a `profiles`) quando o produto exigir rastreio explícito de autor; o trigger de auditoria continua a registar o diff em `audit_logs`.
- **Legado:** tabelas já estáveis podem não ter o pacote completo até evolução controlada.

### Auditoria

- **Função genérica** `log_row_audit()` ligada por trigger às tabelas acordadas; **destino único** `audit_logs` (`entity_type`, `entity_id`, `action`, `old_row`, `new_row`, `changed_by`, `created_at`).

### Indicadores pesados e cargas agendadas

- Agregações muito pesadas (ex.: pré-cálculo para dashboards) podem evoluir para **views materializadas** ou tabelas de snapshot com refresh por job — **fora do núcleo actual**; definir quando o volume o justificar.
- **n8n** (VPS, workflows) mantém-se como automação complementar no roadmap, mas **fica para depois** do núcleo React + Supabase; não entra no critério de aceite das migrações do repositório nesta fase.

# 32. Decisão de escopo final

O sistema completo deve nascer cobrindo os 4 blocos:

```txt

MVP 2: Processo operacional
MVP 3: Painel gerencial
MVP 4: Automação e melhoria contínua
```

Não tratar isso como “futuro indefinido”. Tratar como **plano fechado de execução**, com prioridade para entregar rapidamente o mínimo funcional de cada bloco.

abordagem correta não é esperar perfeição. É:

```txt
implantar rápido
medir rápido
corrigir rápido
expandir com controle
```
